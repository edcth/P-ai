# 消息 JSONL 外置与压缩段分页重构计划

## 1. 当前结论

这次不是“把项目所有读消息的地方改成 JSONL”，也不是“重写所有调用方”。

正确目标是：**替换会话底层存储结构，但保持现有调用方不感知变化**。

现在项目已有统一的会话 JSON shard 读写入口：

```text
conversation_service / commands / runtime
  -> runtime_cache
    -> read_conversation_shard / write_conversation_shard / delete_conversation_shard
      -> chat/conversations/<conversation_id>.json
```

因此第一阶段应该像“升级 `load_json()` 内部实现”一样，在底层读写入口接入新的 message store，而不是让调用方直接读 JSONL。

## 2. 分层边界

目标分层：

```text
业务层：
  conversation_service / commands / model_runtime / tools / archive / remote_im

缓存层：
  runtime_cache
  负责 cached_conversations、dirty ids、mtime、后台写入队列

存储层：
  conversation_storage / message_store
  负责 legacy JSON shard、目录型 JSONL、manifest、index、meta
```

允许依赖方向：

```text
业务层 -> 缓存层 -> 存储层
```

禁止依赖方向：

```text
业务层 -> message_store
业务层 -> JSONL 文件路径
业务层 -> manifest/index/meta 细节
message_store -> 业务层
```

## 3. 硬规则

除存储层自身和缓存层必要接入点外，生产代码不允许直接出现：

```text
message_store::
MessageStore
messages.jsonl
blocks/*.jsonl
messages.idx.json
manifest.json 作为业务判断
```

`runtime_cache` 可以调用一个存储 facade，但不能散落理解 JSONL 细节。

推荐结构：

```text
runtime_cache.rs
  -> read_conversation_storage(...)
  -> write_conversation_storage(...)
  -> delete_conversation_storage(...)

conversation_storage/message_store
  -> legacy <id>.json
  -> directory shard <id>/manifest.json + meta.json + blocks/*.jsonl + messages.idx.json
```

## 4. 非目标

- 不改前端接口。
- 不改命令层调用方式。
- 不改工具调度、模型运行时、归档、审查、远程 IM 的消息读取方式。
- 不让调用方知道 JSONL 存在。
- 不把 `AppData` 立即全量服务化。
- 不在第一阶段移除 `Conversation.messages` 字段。
- 不引入 SQLite。

## 5. 第一阶段最小改动范围

第一阶段只允许改这些边界：

```text
src-tauri/src/features/config/app_data_layout.rs
src-tauri/src/features/core/domain/runtime_cache.rs
src-tauri/src/features/chat/message_store/
src-tauri/src/main.rs include 顺序
```

允许的改动：

- 新增或接入 `message_store` 模块。
- 让 `read_conversation_shard` 支持读取 ready 状态的目录型 JSONL 会话。
- 让 `write_conversation_shard` 在需要时写入目录型 JSONL，或先保持 legacy JSON 写入。
- 让 `delete_conversation_shard` 同时删除 legacy `<id>.json` 和目录型 `<id>/`。
- 让 `app_data_cache_signature` 能感知目录型 shard 的修改时间，避免缓存不刷新。
- 让后台持久化 worker 继续通过原有缓存层写入，不改调用方。

不允许的改动：

- 不把 `commands`、`model_runtime`、`tools`、`archive`、`remote_im` 直接改成调用 `message_store`。
- 不把业务方法里的 `conversation.messages` 全面替换成 JSONL API。
- 不为了 JSONL 改 UI/前端协议。

## 6. 存储读取规则

读取会话时，存储层按显式状态决定来源：

```text
1. 如果目录型 shard 存在 manifest，且 manifest 表示 ready JSONL snapshot：
     读取 meta.json + blocks/*.jsonl，组装成 Conversation 返回。

2. 如果 manifest 不存在，或状态不是 ready：
     回退读取 legacy chat/conversations/<id>.json。

3. 禁止“只要 blocks/*.jsonl 或 messages.jsonl 存在就优先读取”。
```

这可以避免迁移中断、旧 JSONL 残留、回滚后 stale JSONL 抢占真实数据。

## 7. 写入策略

第一阶段写入可以选择保守策略：

```text
默认继续写 legacy <id>.json
```

这样接入后可马上实测读取兼容性，风险最低。

后续再切换为：

```text
如果 manifest 是 ready JSONL snapshot：
  写目录型 JSONL
否则：
  写 legacy JSON
```

写入切换仍然只发生在存储层，调用方不变。

## 8. 删除策略

删除会话时必须同时清理：

```text
chat/conversations/<id>.json
chat/conversations/<id>/
```

删除失败必须记录包含 `conversation_id`、路径和错误信息的日志。

## 9. 缓存签名

当前缓存依赖会话目录下 `.json` 文件的数量、总大小、mtime 等签名。

接入目录型 shard 后，签名必须能感知：

```text
chat/conversations/<id>/manifest.json
chat/conversations/<id>/meta.json
chat/conversations/<id>/blocks/*.jsonl
chat/conversations/<id>/messages.idx.json
```

否则 JSONL 文件变化后，`cached_app_data` 可能误判缓存仍有效。

## 10. 迁移状态

Manifest 至少表达：

```text
storeKind: legacyJson | jsonlSnapshot | jsonlEventLog
migrationState: notStarted | building | ready | failed | rolledBack
sourceMessageCount
lastMessageId
messagesJsonlBytes
messagesIndexRevision
version
updatedAt
```

只有：

```text
storeKind = jsonlSnapshot
migrationState = ready
```

才允许作为读取真相。

## 11. JSONL 语义

当前优先采用“快照 JSONL”：

```text
每行是一条完整 ChatMessage 快照
index 只记录 message_id、block_id、offset、byte_len
index 禁止写入对话正文、正文预览、工具参数正文或 provider 原始 JSON
压缩边界由 `blockId` 切换表达，不再额外写 `compactionKind`
```

暂不采用事件日志作为第一阶段真相，因为现有业务存在撤回、截断、替换、压缩、分支、归档等非纯追加操作。

## 12. ConversationBlock 物理切块

新的消息层级固定为：

```text
Conversation
  -> ConversationBlock
    -> Message
```

目录布局：

```text
chat/conversations/<conversation_id>/
  manifest.json
  meta.json
  messages.idx.json
  blocks/
    000000.jsonl
    000001.jsonl
    000002.jsonl
```

切块规则：

- 压缩/摘要消息作为新 `ConversationBlock` 的第一条消息。
- 一个 block 表示“从上一次摘要恢复开始，到下一次摘要前结束”的稳定上下文周期。
- 普通会话只按压缩/摘要边界切块。
- 联系人会话的迁移/全量重建按“压缩/摘要边界或本地 04:00 日界线”切块；从最新消息所属会话日开始向前整理历史，避免旧联系人历史因为只有一个压缩标记而落入单个大块。
- 联系人无压缩按天切块时，04:00 前的消息归入前一天，避免凌晨连续对话被 0 点硬拆。
- 联系人会话的正常增量写入只延续/更新尾块；跨过 04:00 日界线或新增压缩/摘要边界时才新增 block，并触发旧 block 瘦身。
- 按天切块只针对联系人会话，不影响普通会话。
- `messages.idx.json` 是全局索引，索引项必须携带 `blockId`，offset/byteLen 只相对所属 block 文件有效。
- 旧单文件 `messages.jsonl` 仅作为兼容读取路径，不再作为新写入目标。

瘦身规则：

- 最新 2 个 `ConversationBlock` 原样保留，保证当前调度、工具结果和前台 UI 不受影响。
- 更早的 block 保留文本消息和所有外置/URL 媒体引用，确保 LLM 回看历史时仍知道用户发过哪些图片、PDF、音频或文件。
- 更早的 block 丢弃内嵌 base64 媒体，避免旧历史重新撑大 JSONL。
- 更早的 block 会丢弃工具调用、MCP 调用、额外上下文挂件、运行态 provider meta。
- 压缩/摘要边界属于结构信息，必须保留最小 `message_meta.kind`，否则后续重写会丢失切块边界。
- 日常无感更新的最佳触发点是“新增 block 时”：新 block 出现后，原来的倒数第二个 block 变成倒数第三个，此时才把这个刚稳定的 block 瘦身。
- 已经稳定的旧 block 不应每次无脑重写，也不应为了“比较是否变化”而读取旧 block 内容。
- 日常 ready 写入只能读取 manifest/index 元数据；普通写入只重写最新 block，新增 block 时额外重写刚进入历史区的倒数第三个 block。
- 只有迁移/结构性重建允许全量生成所有 block。

## 13. 阶段计划

### 阶段 1：底层接入，可立即实测

- include `message_store`。
- `read_conversation_shard` 优先读取 ready JSONL snapshot，否则回退 legacy JSON。
- `delete_conversation_shard` 清理 legacy 文件和目录型 shard。
- `app_data_cache_signature` 感知目录型 shard。
- 写入先保持 legacy JSON，保证风险最小。
- 跑 `cargo check`。
- 手动实测现有会话读取、切换、发送、删除不变。

### 阶段 2：写入切换

- 如果会话已经是 ready JSONL snapshot，则 `write_conversation_shard` 写回目录型 ConversationBlock JSONL。
- legacy 会话继续写 `<id>.json`。
- 后台 persist worker 不改调用方式。
- 增加写入/读取往返测试。

### 阶段 3：显式迁移工具

- 增加单会话迁移方法：legacy JSON -> directory ConversationBlock JSONL snapshot。
- 迁移必须独立可测。
- 中断/失败状态继续读取 legacy JSON。
- 不自动批量迁移全部会话。

### 阶段 4：按需优化读取

- 在不改调用方的前提下，逐步让已有缓存/会话服务方法利用 message_store 索引做分页、最近消息、按 id 读取优化。
- 仍然禁止业务层直接访问 `message_store`。

### 阶段 5：AppData 去消息正文化

- 等底层存储稳定后，再考虑让 `AppData.conversations` 逐步退化为轻量 meta/index。
- 这不是第一阶段目标。

## 14. 测试要求

第一阶段最低验证：

- `cargo check --manifest-path src-tauri/Cargo.toml`
- 旧 `<id>.json` 会话仍能读取。
- 存在 ready ConversationBlock JSONL directory shard 时能读取。
- 新写入会生成 `blocks/*.jsonl`，不是单个 `messages.jsonl`。
- `messages.idx.json` 的 `blockId`、offset、byteLen 能定位到正确消息。
- 更早 block 会被瘦身，最新 2 个 block 保持完整。
- building/failed/stale manifest 不会抢占 legacy JSON。
- 删除会话同时删除 legacy 文件和目录。
- 缓存签名能感知目录型 shard 修改。

边界检查：

```powershell
rg -n "message_store::|MessageStore|messages\.jsonl|messages\.idx\.json" src-tauri\src |
  Select-String -NotMatch "features\\chat\\message_store|features/chat/message_store|features\\core\\domain\\runtime_cache|features/core/domain/runtime_cache|features\\config\\app_data_layout|features/config/app_data_layout"
```

## 15. 当前修正结论

之前大范围修改调用方是错误方向。

正确方向是：

```text
保持调用方稳定；
保持 conversation_service 基本稳定；
让 runtime_cache 继续负责缓存；
只把 runtime_cache 下面的会话存储读写从 legacy JSON 扩展为 legacy JSON / JSONL 双后端。
```

也就是：不是全项目改成 `load_jsonl()`，而是把原来的 `load_json()` 内部升级为能识别新存储格式。
