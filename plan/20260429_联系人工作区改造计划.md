# 联系人工作区改造计划

## 背景

当前远程联系人会话的工作区模型与普通会话存在差异：

- 联系人配置里没有工作区字段，无法为联系人指定专属目录
- 联系人会话的系统目录始终是 full_access，不符合"联系人只能动自己桌面"的语义
- 主部门联系人走主会话、非主部门走独占会话的旧残留语义需要收口
- 联系人会话的工作区编辑入口与普通会话不一致

本次改造要让联系人拥有与普通会话同构的工作目录能力，但配置入口放在联系人设置卡，运行时系统目录强制只读。

## 目标

- 每个联系人可配置一套完整工作区列表（主目录 + 多个副目录，保留 access 配置）
- 联系人会话运行时动态投影有效工作区，系统目录强制 read_only
- 联系人自定义目录按其配置自然生效（full_access / approval / read_only）
- 联系人会话保持独立类型，不可归档、工具分配特殊
- 联系人统一使用独立联系人会话语义，收口旧 route_mode / 主会话残留

## 已确认规则

1. 联系人会话是独立会话类型，不等同普通会话，保持 remote_im_contact 类型不变
2. 联系人会话的工作区配置复用 ShellWorkspaceConfig 结构，与普通会话完全一致
3. 联系人工作区配置入口只在联系人设置卡，联系人会话内禁用"设置会话工作目录"
4. 联系人配置的工作区是真源，联系人会话运行时动态投影，不依赖会话里旧的 shell_workspaces
5. 联系人会话有效工作区组装规则：
   - 系统目录固定注入，但强制 read_only
   - 再合并联系人自定义工作区列表
6. 不强制联系人必须配置 main，未配置时允许只读运行，不额外报错
7. 联系人会话不可归档、联系人工具注入规则不变
8. 前后台不再区分，联系人统一按独立联系人会话语义理解

## 实施方案

### 1. 联系人工作区数据模型

**后端改动**

- `src-tauri/src/features/core/domain/types_storage.rs`：为 `RemoteImContact` 新增 `shell_workspaces: Vec<ShellWorkspaceConfig>` 字段
- 兼容旧数据：serde `default` 使其反序列化时默认为空列表
- 更新联系人创建/更新相关命令，支持保存工作区配置

**前端改动**

- `src/types/app.ts`：同步 `RemoteImContact` 类型，新增 `shell_workspaces` 字段

### 2. 联系人设置卡 UI

**文件位置**

- `src/features/config/views/config-tabs/RemoteImTab.vue`

**改动内容**

- 在联系人设置模态中新增"工作目录"配置区
- 复用普通会话工作区配置的结构和交互：
  - 主目录选择（level = main）
  - 副目录列表（level = secondary）
  - 每个目录的 access 配置（full_access / approval / read_only）
  - 增删改操作
- 保存时调用后端命令更新联系人的 shell_workspaces

### 3. 联系人会话工作区投影

**核心逻辑位置**

- `src-tauri/src/features/system/tools/terminal/workspace.rs`：`terminal_allowed_workspaces_for_conversation_canonical`

**投影规则**

当会话类型为 remote_im_contact 时：
1. 查找该会话绑定的联系人（通过 bound_conversation_id 反查或会话上下文）
2. 取联系人的 shell_workspaces 作为工作区真源
3. 组装有效工作区列表：
   - 系统目录固定注入，但 access 强制为 `read_only`
   - 联系人自定义工作区按原 access 配置合并
4. 若联系人未配置任何工作区，则只有 system/read_only

**实现路径**

- 在 `terminal_allowed_workspaces_for_conversation_canonical` 中增加 remote_im_contact 分支
- 或在上游先组装好联系人会话的有效工作区，再传入现有逻辑
- 优先选择侵入最小、逻辑最清晰的路径

### 4. 终端 / 补丁 / AGENTS 适配

**终端 exec**

- `src-tauri/src/features/system/tools/terminal/exec.rs`
- 复用现有权限判断链，只要联系人会话拿到正确的工作区列表，现有规则自然生效
- 系统目录 read_only → 只读白名单命令仍可读，写命令被拦截

**补丁 apply_patch**

- `src-tauri/src/features/system/tools/patch.rs`
- 同上，复用现有权限判断，工作区列表正确即可

**AGENTS 注入**

- `src-tauri/src/features/system/commands/prompt_assembly.rs`
- `conversation_user_main_workspace_root` 只看 main workspace
- 若联系人配置了 main 目录，AGENTS.md 自然注入
- 若未配置，AGENTS 不注入，这是预期行为

### 5. 旧联系人会话语义收口

**收口范围**

- 移除"主部门联系人走主会话"的残留逻辑
- 联系人统一使用独立联系人会话语义
- `route_mode` 语义弱化，不决定联系人进主会话还是独占会话

**保持不变**

- 联系人会话类型仍为 remote_im_contact
- 联系人会话不可归档
- 联系人工具注入规则不变
- 联系人来源链路不变
- 只改与工作区投影直接相关的最小范围

- `src-tauri/src/features/system/tools/patch.rs`：补丁权限判断
- `src-tauri/src/features/system/commands/prompt_assembly.rs`：AGENTS 注入
- 以及必要的会话服务、联系人服务相关文件

## 验证范围

1. 联系人设置保存：配置工作区后保存，重启后配置仍在
2. 联系人会话工作区展示：聊天窗口工作区面板显示联系人配置的工作区
3. exec 写权限：联系人自定义 full_access 目录可写，系统目录写命令被拦截
4. apply_patch 写权限：同上
5. 系统目录只读拦截：read_only 目录下的写操作被拒绝
6. main 工作区 AGENTS 注入：联系人配置 main 目录后，AGENTS.md 自然注入
7. 未配置场景：联系人未配工作区时，只有 system/read_only，能读不能写

## 风险点

1. **联系人历史会话迁移**：老联系人没有 shell_workspaces 字段，老会话可能已持久化旧工作区。采用运行时覆盖优先，避免危险迁移。
2. **主部门联系人收口**：主部门联系人从走主会话改为走独立联系人会话，需确认会话创建/绑定链路是否需要调整。
3. **文件发送工具**：contact_send_files 需要真实本地路径，系统目录只读不影响文件发送能力，但需验证。
