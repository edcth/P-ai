# 20260425_会话内审查面板与侧边栏收口计划

## 背景

当前终端审批仍采用全局模态弹窗，触发后会直接抢占当前聊天视图。用户已经明确要求把审查体验收窄为会话内交互：输入区切换为审查面板，详情仍复用现有审查侧边栏，不再让全局弹窗压住别的会话。

同时，现有工具审查侧边栏已经具备按 `conversationId` 读取批次、按批次查看详情与提交最终审查的能力，因此本轮应优先复用既有结构，而不是重做消息流内卡片。

## 已核实现状

### 1. 终端审批当前是全局队列 + 全局模态

- 前端 `use-terminal-approval.ts` 目前只维护 `queue[0]` 作为当前审批项，`terminalApprovalDialogOpen` 由队列非空直接推出。
- `ShellDialogsHost.vue` 固定挂载 `TerminalApprovalDialog`，因此任一审批请求到来都会以全局模态方式覆盖当前界面。
- `UnifiedWindowApp.vue` 通过 `easy-call:terminal-approval-request` 事件统一入队，没有额外的会话内显示分流。

### 2. 审批请求本身已经带会话归属线索

- Rust 侧 `terminal_request_user_approval(...)` 发出的 payload 已包含 `sessionId`。
- 这意味着前端无需先改后端协议，也能把审批项绑定回对应会话；缺口主要在前端状态归属和显示方式。

### 3. 工具审查侧边栏已经是按会话读取的详情容器

- `use-chat-tool-review.ts` 已按 `activeConversationId` 调 `list_tool_review_batches`、`get_tool_review_item_detail`、`submit_tool_review_batch`。
- `ChatView.vue` 当前通过 `toolReviewPanelOpen` 挂出 `ToolReviewSidebar`，它天然是会话内、随聊天页切换的。
- 因此“查看详情按钮 -> 打开侧边栏”可以直接复用既有 `toggleToolReviewPanel()` 能力。

### 4. 输入区是最合适的审查面板落点

- `ChatView.vue` 已把输入区集中在 `composerContainer` 内，由 `ChatComposerPanel` 承担发送、录音、附件等交互。
- 在这里根据“当前会话是否有待审批项”切换为审查面板，可以天然阻塞当前会话输入，而不会影响别的会话浏览。

## 目标

把“终端审批全局模态”改为“当前会话输入区审查面板 + 工具审查侧边栏详情”模式：

1. 审批请求只归属其 `sessionId` 对应的会话。
2. 仅当前激活会话存在待审批项时，输入区覆盖为审查面板。
3. 审查面板以列表展示当前会话待审批项摘要，并提供整项 `批准 / 驳回 / 查看详情` 操作。
4. “查看详情”只负责打开现有工具审查侧边栏，不再拉起全局模态。
5. 切走会话时不显示其他会话的审批面板；切回时若仍有待审批则恢复显示。

## 非目标

- 不把审批内容塞进消息滚动区。
- 不新增第二详情容器或新的审查弹窗。
- 不改造为逐卡审阅工作流。
- 本轮不追求批量合并多个终端审批为单次统一提交；先保持逐项批准/驳回，但显示层收口到会话输入区。

## 实施方案

### 一、把终端审批队列从“全局当前项”改为“按会话派生显示”

1. 扩展前端 `TerminalApprovalRequestPayload` 的归属语义，统一把 `sessionId` 视作 conversation/session 绑定键。
2. 在 `use-terminal-approval.ts` 中保留原始队列，但新增派生能力：
   - 当前激活会话的待审批列表。
   - 当前激活会话的首个待处理审批项。
   - 判断当前会话是否处于审批态。
3. 审批提交时仍按 `requestId` 调 `resolve_terminal_approval`，只处理当前项，保持后端最小改动。

### 二、移除全局模态主入口，改成聊天输入区审查面板

1. 停止在 `ShellDialogsHost.vue` 中渲染 `TerminalApprovalDialog`。
2. 在 `ChatView.vue` 的 `composerContainer` 内增加 `ChatApprovalPanel` 一类新组件，替代输入框显示：
   - 条件：当前激活会话存在待审批项。
   - 否则仍显示原 `ChatComposerPanel`。
3. 审查面板内容：
   - 标题与数量提示。
   - 当前会话待审批项列表。
   - 每项显示工具名、摘要、风险提示、顺序。
   - 每项提供 `批准`、`驳回`、`查看详情`。

### 三、把“查看详情”收口到现有工具审查侧边栏

1. 审查面板点击“查看详情”时：
   - 若侧边栏未打开，则调用现有 `toggleToolReviewPanel()` 打开。
   - 视情况选中当前 batch；若缺少终端审批与工具审查批次映射，首版只负责打开侧边栏，不强做精确定位。
2. 保持 `ToolReviewSidebar` 继续挂在 `ChatView.vue` 右侧，作为详情页，不再承担主入口抢占。

### 四、会话切换与阻塞规则

1. 输入区审查面板只看当前 `activeConversationId` 对应的审批项。
2. 切换会话时：
   - 非当前会话的审批项不显示。
   - 输入区恢复或切换为该会话自己的状态。
3. 审批态只阻塞当前会话输入；其他会话可以切换浏览。

## 风险与处理

### 1. `sessionId` 与 `activeConversationId` 可能并非完全同义

需要先核对当前终端审批发起场景里，`sessionId` 是否稳定对应聊天会话 ID。若存在不一致，必须先在前端做兼容映射；证据不足前不做后端协议扩展。

### 2. 工具审查侧边栏与终端审批详情并不完全同源

终端审批项与工具审查批次不一定一一对应，因此“查看详情”首版可能只能做到打开侧边栏，而不能保证精确定位到某一项。若调查发现无稳定映射，就明确接受这个首版限制。

### 3. 全局模态移除后，非聊天窗口不再可见终端审批

当前 bootstrap 已只在 chat window 监听审批事件，因此本轮方案仍以聊天窗口为主，不额外扩展到 archives/config 窗口。

## 验证范围

1. 当前会话出现终端审批时，输入区变为审查面板，不再弹全局模态。
2. 切换到其他会话时，该审批面板隐藏。
3. 切回原会话时，审批面板恢复。
4. 点击批准/驳回后，能正确调用后端 resolve 并从列表移除。
5. 点击查看详情后，能打开现有工具审查侧边栏。
6. 前端类型检查通过。

## 实施顺序

1. 完成前端状态归属改造，确认当前会话审批项可正确派生。
2. 新增输入区审查面板，并接入 `ChatView.vue` 切换逻辑。
3. 移除全局模态挂载，收口到聊天页。
4. 打通“查看详情 -> 打开侧边栏”。
5. 做最小验证并补记录。
