# 终端与补丁审批链路重构归档（2026-04-16）

## 1. 背景

本轮目标是把终端命令与 `apply_patch` 的审批体验从“勉强可用”收口到真正可审、可读、可终止：

- 审批卡片要能明确告诉用户风险与影响范围
- `apply_patch` 审批不能再展示空黑框、假行号、错误示例
- 审批前应先校验补丁是否真的可应用
- 用户拒绝后，本轮调度应立刻终止，而不是继续让模型想下一步

用户已在实际界面上连续验证多轮，并在本轮结束时明确反馈“完美”，因此按本归档收口。

## 2. 实际落地

### 2.1 审批弹窗前端

前端审批弹窗已完成以下调整：

- 新增独立的 `TerminalApprovalImpactPanel`
- 新增独立的 `TerminalApprovalPatchSample`
- 影响范围改为列表，不再使用一项一个小卡片
- 影响范围会明确标记 `新增 / 更新 / 删除`
- 风险说明卡只保留风险胶囊着色，不再额外染底色或改正文颜色
- 操作按钮改为居中布局，并统一为“批准 / 拒绝”

终端命令文本现在统一进入 `mockup-code` 预览区域显示，不再悬浮在风险说明上方。

### 2.2 `apply_patch` 预览规则

补丁预览最终收口为：

- 只展示真实的 `+ / -` 差异内容
- `Update File`、`Add File`、`Delete File` 都允许展示 diff
- 仅当当前补丁块没有任何 `+ / -` 内容时，才不渲染 code 区
- 多个补丁块按分页切换展示

这部分已经移除之前的静态样板和调试入口，正式界面只展示真实审批数据。

### 2.3 审批前 dry-run 预检

`apply_patch` 链路已改为：

1. 先解析补丁
2. 再解析路径
3. 再对补丁做 dry-run 预检
4. 只有预检通过才进入审批
5. 审批通过后再正式写盘

其中 `Update File` 会在审批前真实读取目标文件并匹配 hunk。

如果 hunk 上下文不匹配，会直接报错，不再弹出审批。

### 2.4 真实行号预览

审批预览不再依赖模型原始补丁中的 hunk 行号。

后端现在会根据 dry-run 的真实匹配位置重新生成审批预览：

- `Update File`：使用真实匹配到的旧/新行号生成 `@@ -a,b +c,d @@`
- `Add File`：从新增文件的首行开始生成
- `Delete File`：基于原文件真实内容生成删除 diff

因此前端展示的行号来源已经从“模型声称的行号”切到“后端确认后的真实行号”。

## 3. 调度终止语义

本轮还修正了审批拒绝后的控制流。

现在当用户拒绝 `exec` 或 `apply_patch` 时：

- 拒绝结果会写入工具历史
- 当前轮会生成一条正式助手消息
- 本轮调度立即终止

不再发生“工具被拒绝，但模型继续拿拒绝结果往下推”的行为。

## 4. 与原始状态相比的关键提升

本轮确认收口的关键提升包括：

1. 审批不再超时，用户可以一直等到看完再决定。
2. 审批卡片只保留有效信息，移除了工作目录、超时提示等噪音。
3. `apply_patch` 预览不再依赖错误样板，而是只展示真实补丁内容。
4. 根本不可应用的补丁不再进入审批。
5. 审批拒绝会真正终止当前轮，而不只是返回一个普通工具结果。

- 真实 diff 预览与真实行号生成
- 审批拒绝后的工具历史落盘与本轮终止
